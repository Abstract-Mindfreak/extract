## 🎯 Контекст и Ограничения
- **Железо**: Локальная машина, 16 ГБ ОЗУ, слабая GPU. Критически важна экономия памяти.
- **Модель эмбеддингов**: Локальный Ollama, модель `embeddinggemma:300m` (http://localhost:11434).
- **База данных**: PostgreSQL с уже установленным расширением `vector`. Две отдельные базы данных (или схемы), содержащие данные в формате `jsonb`.
- **Цель**: Добавить возможность векторизации существующих JSON-промтов и всех данных необходимых для работы Flowmusic Agents и реализовать семантический поиск (RAG) через UI в ASE Console.

## 📋 Этапы реализации (Адаптируй под проект!)

### Этап 0: Анализ и Подготовка (ОБЯЗАТЕЛЬНО ПЕРВЫЙ ШАГ)
1. Найди в коде модели/схемы БД, соответствующие таблицам с `jsonb` промтами в обеих базах.
2. Определи текущий стек бэкенда (Python/Node.js/Go?) и фронтенда (React/Vue/другое?) для ASE Console.
3. Напиши и выполни небольшой скрипт/запрос, который отправляет тестовый текст в `embeddinggemma:300m` через Ollama, чтобы **динамически узнать размерность вектора** (например, 768, 1024 или 2304). Не хардкодь это число!

### Этап 1: Модификация схемы БД
Для каждой целевой таблицы с промтами создай связанную таблицу (или добавь колонки, если это уместно в текущей архитектуре) для хранения эмбеддингов:
- Ссылка на оригинальную запись (`prompt_id` или аналог).
- Текстовое представление промта (`chunk_text`), из которого был получен вектор.
- Колонка `embedding vector(N)`, где `N` — размерность, полученная на Этапе 0.
- **Индекс**: Создай индекс `ivfflat` с оператором `vector_cosine_ops` для быстрого поиска на CPU.
- (Опционально) Добавь булевый флаг `is_vectorized` в основную таблицу для отслеживания прогресса.

### Этап 2: Пайплайн пакетной векторизации (Backend)
Создай скрипт или фоновую задачу (background job) для заполнения векторов:
- Используй пакетную обработку (batch size = 10-20 записей), чтобы избежать переполнения 16 ГБ ОЗУ.
- Извлекай текст из `jsonb` (конкатенация ключевых полей или весь JSON как строка).
- Отправляй пакет в Ollama (`embeddinggemma:300m`), получай векторы и сохраняй их в БД.
- Реализуй обработку ошибок (если Ollama недоступен, задача должна безопасно падать или повторяться).

### Этап 3: Логика RAG-поиска (Backend)
Реализуй эндпоинт или функцию `search_rag(query: str, top_k: int = 3, db_identifier: str)`:
1. Конвертирует `query` в эмбеддинг через Ollama.
2. Выполняет PostgreSQL запрос с оператором косинусного расстояния `<=>` для поиска `top_k` наиболее похожих записей.
3. Возвращает найденные `jsonb` данные вместе с оценкой схожести (distance).

### Этап 4: Интеграция в UI (ASE Console / Flowmusic Agents)
Используя существующий фронтенд-фреймворк проекта, добавь в интерфейс:
1. Кнопку "Векторизовать базу данных" (Trigger для Этапа 2). Желательно с индикатором прогресса или статусом задачи, чтобы не блокировать UI.
2. Поле "Умный поиск" (Smart Search), которое отправляет запрос на эндпоинт из Этапа 3 и красиво отображает найденные JSON-промты пользователю.
